Performance, what is jank?

Notes
* Jank is when the frame takes longer to layout and render than 16ms. Causes a stutter it feels bad. * Caused by slow synchronous events * e.g. XHR, big localstorage uses but most commonly... * Page layouts and paints -------- * Layout is slowest * Making a small DOM change could potentially invalidate the entire DOM tree causing it to be recalculated * An invalidated DOM is recalculated each frame and whenever a layout property is read -------- * Paint is medium slow * This is where the layer's bitmap has been invalidated and needs to be redrawn and stored in memory * This is done on each render -------- * Composite is very fast * Composite only involves the graphics card rerendering from bitmaps calculated in paint

Bad Stuff

Notes
* Two commonly seen mistakes * The First is Reading then writing in a loop which thrashes the DOM becasue the layout is invalidated and recalculated on every single read. * The second is animating properties which cause layouts on the DOM this slide is making my fan spin up and makes a phone cry and this has a relatively simple DOM structure. On this slide the layout is recalculated every single frame. ----------- * The fix for the first situation tends to be fairly simple batch your reads and write all together. Don't change a property then immediately read it. Don't append to an `innerhtml` in a loop add it to a seperate string and update it in one go. * Wilson Page's fastdom library can help with this in an mvc or a large framework with many independent modules so that reads and writes in a single frame all get done together. * A good general motto is to calculate all DOM changes first then apply them in a single step. ----------- * This still leaves us with the second issue, how do you animate something which changes the layout? * One very common use case is a list widget which grows as an element is added. * This can affect the whole dom so would be very expensive to animate.

All is not hopeless...

Notes
* It is possible to alter the layout on a simple widget and not have the whole dom affected. * There is a draft css spec called containment which will hint to the browser in much the same way as the will-change property that this element will not affect the rest of the DOM tree. * It requires cerain properties such as no scrolling and fixed dimensions. * It could be done in browsers previouly but it relied on the element having certain properties before it would have layout boundaries such as a fixed height and no scroll, the containment spec still requires these properties but makes it clear what is happening which is good.
